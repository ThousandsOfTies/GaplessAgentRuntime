## 出発点 — Gapless Agent Runtime の基礎機能

すべての出発点はここにある。

> **Gapless Agent Runtime の基礎機能は、「複数の環境に同時に接続し、その上で AI を縦横無尽に
> 稼働させること」。**

ビルド環境・シミュレーション環境・実機環境という性質の異なるリソースに、エージェントが
同時につながり、どこに対しても同じ操作（接続・取得・反映＝ put / get）でシームレスに
振る舞える。環境の違い（プロトコル・場所・物理／仮想）を隠蔽する「統一された I/O 平面」を
提供しているのが本質。人間がそれぞれの環境を手で橋渡ししていた構図（Before）から、
エージェントが中核となって各環境を貫く構図（After）への移行——これが基礎機能の正体。

## いま立っている場所 — 組み込み機器開発という PoC

現状、この基礎機能の **最初の適用先（ターゲットユースケース）** として、組み込み機器開発を
選んで PoC を進めている。

- **ビルド環境** … ファームウェア・制御コードを書く
- **シミュレーション環境** … CUSE / gpio-sim による仮想デバイス（実機が無くても動く）
- **実機環境** … Raspberry Pi 等のボード上の実デバイス

この三層にエージェントが同時接続し、「仮想で予行 → 確信してから実機」を回す。
gpio-sim 移行で進めているのは、まさに「実機とシミュレータを同じ操作で扱える透過化」だった。

## 視点を上げる — 基礎機能は組み込みに固有ではない

ここで一段、抽象度を上げる。「**ビルド／シミュレーション／実機**」という三層構造は、
実は組み込みに固有ではなく、**多くの分野に同じ形（アイソモーフィック）で存在する**。
三層が揃っている分野なら、Gapless Agent Runtime の基礎機能はそのまま移植できる。

| 分野 | ビルド | シミュレーション | 実機 |
|---|---|---|---|
| ロボティクス | 制御ポリシー | Gazebo / Isaac Sim | ロボット本体 |
| 製造（インダストリ4.0） | 生産レシピ・PLC | デジタルツイン | 産業機械・ライン |
| クラウド／インフラ | IaC コード | plan / staging | 本番インフラ |
| 自動車 | ECU ソフト | SIL / HIL リグ | 車両 |
| ヘルスケア・創薬 | プロトコル・モデル | in-silico | 実験装置・臨床 |
| 宇宙 | フライトソフト | デジタルツイン・地上テストベッド | 軌道上の機体 |

気づくのは、**新しさは「分野」ではなく「三層の境界をどう透過させるか」に宿る**こと。
そして移植価値が高いのは——**「実機が高い・危ない・遅い・取り返しがつかない」× 「人が
環境間を手で橋渡ししていた」** 分野ほど大きい。

## 選んだシナリオ — 宇宙とロボット

数ある適用先の中で、エンジニアとして最も燃えるのは **宇宙** と **ロボット**。
どちらも「実機が高い・危ない・遅い」が極端に効くため、基礎機能の価値が最も劇的に出る。

### 宇宙 — put / get が「生命線」になる世界

宇宙が際立つのは、**打ち上げた瞬間、実機にもう物理的に触れられない**という制約。
人間ハブ（Before の図）が物理的に不可能になり、機体との唯一の接点が
**テレメトリ（get）／テレコマンド（put）だけ**になる。

- 「必ず sim で試してから実機に put」が任意ではなく **設計要件レベルで強制** される
- 通信遅延・断続的な可視時間（地上局パス）＝ **非同期 put/get・イベント駆動 get** が現実問題に
- → Gapless Agent Runtime の抽象が「あったら便利」ではなく **「無いと運用が成立しない」** 領域

### ロボット — put / get ループが「加速装置」になる世界

ロボットの強みは逆に、**ループを速く回せる**こと。sim2real を何百回も自律で回して
学習・是正できる。エージェント常駐（After の図）が最も活きるのはこちら。

> **宇宙 ＝ put/get が生命線（透過性が必須要件）**
> **ロボット ＝ put/get ループが加速装置（自律反復で価値）**

### そして、宇宙 ＋ ロボット ＝ ガンダム

この二つの制約が同時に最大化する究極のユースケースが、象徴的に言えば「ガンダム」。
コックピット（人間が put/get する場所）と、学習型コンピュータ（常駐エージェント）が
一つの機体に同居し、「人間ハブ（Before）から機体自律（After）へ、どこまで移譲するか」
というせめぎ合いそのものを描いている。これは今 gpio-sim でやっている
「人間の手数を減らしてエージェントを実機に直結させる」流れの、思想的な極北にあたる。

## 着地 — 夢を「回せるサイズ」に畳む

とはいえ、宇宙も重機も **いきなりは回せない**（1 回の試作が高すぎる・危険すぎる）。
そこで、手法を確立する「実験室」として **10〜20cm のおもちゃサイズ** にスケールを畳む。
ここから先が、具体的な研究構想——「遊びを通じた最適デバイスの発見」になる。

## 一言で

**シミュレーション環境の中に「まだ存在しないデバイス」を先に捏造し、
エージェントが遊び（対戦・タスク）の中で試行錯誤するフィードバックループを回して、
「こんなデバイスが現実にあれば役立つ／面白い」という “ハードウェア要求仕様” を逆算して発見する。**

## 普通の sim2real と矢印が逆（real-from-sim / sim2spec）

| 普通の sim2real | この構想（real-from-sim） |
|---|---|
| 実機が先にある → sim で模倣・検証 | **sim が先にある → 実機を「要求仕様」として生成** |
| シミュレーションは検証の道具 | シミュレーションは **発明の探索空間** |
| 出力＝検証済みの制御 | 出力＝**「作ってほしいデバイスの仕様」** |

ポイントは「シミュレーションを、デバイスを発明するための探索空間として使う」こと。
FB ループの出力がハードウェア要求仕様になる。

## なぜ今の PoC 資産がそのまま土台になるか

| 既存資産 | この構想での役割 |
|---|---|
| `agp-tools/cuse-stubs/`（cuse_gpio / cuse_i2c / cuse_spi） | **存在しないデバイスを OS に生やす捺造装置** |
| `vl53l0x_sim.c` / `mfrc522_sim.c` / `ssd1306_sim.c` | デバイスの振る舞いをコードで定義 ＝ 新デバイスも同じ枠で「発明」できる |
| `web-bridge/` | 人間が観察・介入する窓（コックピット） |
| `scenarios/*.json` | ループの初期条件を宣言的に与える仕組み |

→ 「架空デバイスを 1 個 `.c` で書く → CUSE で OS に生やす → エージェントが get/put で叩いて
タスクさせる → 上手くいかない／面白くないを FB して仕様を進化させる」という閉ループが、
今の構造の延長で組める。

## ループの形

```
架空デバイス仕様 ──→ CUSE で仮想デバイス生成 ──→ タスク sim（エージェントが put/get）
      ↑                                                      │
      └──────────── FB（面白い? 強い? 物理的に妥当?）←───────┘
                              │
                      収束 ──→ 現実への提案「このデバイス作って！」
```

評価軸（D）を何にするかで研究の色が決まる:
1. **ゲーム性** … タスク／対戦が面白いか → エンタメ・ゲーム AI
2. **物理的実現可能性** … その架空デバイス、現実に作れるか → ここを入れると「妄想」が「提案」に化ける
3. **学習効率** … その新デバイスがあると学習が速いか → ロボ・強化学習

## 発散を防ぐ制約（研究として締めるための芯）

「何でもアリ」を避けるため、制約を意図的に入れる:

- **スケール制約**: ターゲットは **10〜20cm のおもちゃサイズ**。
  - 安い・速い・安全 → FB ループを何百回も回せる実験室になる。
  - 3D プリント＋ホビー部品で **実機化が現実的**。
  - 「実機が高い・危ない・遅い」を意図的に外したスケール（宇宙／重機の真逆）。
- **物理制約**: 架空デバイスも実在の物理法則（応答速度・消費電力・ノイズ）を守る → 出力が「作れる仕様」になる。
- **インターフェース制約**: 新デバイスも GPIO / I2C / SPI のいずれかで喋る → 既存 cuse-stubs でそのまま実機化の道が繋がる。

## 機体の設計判断（対話で固めた骨格）

| 項目 | 決定 | 理由の要点 |
|---|---|---|
| スケール | 10〜20cm おもちゃ | 安全・高速・実機化可能な実験室 |
| 機体 | 3 足・**正三角形 120° 等配** | 静的安定◎。3 回回転対称で前後がなく全方向性。形状由来の交絡が消え、架空デバイスの効果が純粋に評価へ乗る |
| 脚の自由度 | **2DOF / 脚（計 6 サーボ）** | 軽い・安い・収束が速い。**「良い欠乏」** が架空デバイスへの要求を引き出す（膝が無いから「膝の代わり」を欲しがる） |
| 初期タスク | **立ってモノ運び** | 評価が数値で一意。架空デバイスの効果が因果で読める。3 足の静的安定が得意な領域 |
| 進行 | カリキュラム（立つ→運ぶ→競争→妨害→殴り合い） | きれいな評価信号で手法を固めてから、相互作用の濁りに進む |
| 手法 | real-from-sim（捏造→FB ループ→仕様逆算） | — |

### 設計の核となった洞察

- **「良い欠乏」**: 機体は素のままだと少し困るくらいが良い。何でもできると架空デバイスを発見する動機が生まれない。「あと一歩届かない」状況がエージェントに「こういうセンサ／関節があれば…」と要求させ、発見ループを駆動する。
- **3 足の非対称性**: 「立つ＝超得意／歩く＝急に難しい」が同じ機体に同居。カリキュラムと噛み合う。3 足歩行は自然界にほぼ無く、生物模倣に答えがない＝本当に「発見」になる。
- **120° 等配の二面性**: 立つ・運ぶは全方向に最強／歩くと 3 回対称の「歩行方向ムラ」と戦う。これがまた得意と難問の同居を生む。

## スケーラビリティ仮説（なぜおもちゃから始めるか）

```
おもちゃスケール(10-20cm)        同じ FB ループ構造        重機・産業スケール
 安い・速い・安全     ──手法を確立──→     ──スケール則──→    高い・遅い・危険
      └──────────────── 検証の母体 ────────────────────────┘
```

- おもちゃ＝手法を確立する「実験室」（ループを回しまくって方法論を固める）
- 重機＝その手法の「応用先」（回せない領域に real-from-sim を移植）
- 最終的な研究の出力像: **「アクチュエータを増やす vs センサで賢くする」の最適点の自動発見。**
  これは「重機レベルでのルール最適デバイス発見の自動化」の縮図。

## 横たわる論点（自律を上げるほど効いてくる）

- **put の安全境界**: 自律を上げるほど「誰が引き金を引くか」が問題になる。おもちゃの LED を光らせる安全境界と、地続き。
- **一貫性／同期・イベント駆動の get**: 複数接続先への同時 put の原子性、リアルタイム性。
- これらは前提条件であり、構想を実装に落とす段になって初めて本気で考える。
