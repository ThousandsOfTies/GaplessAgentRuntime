# 設計思想 — なぜ仮想と実機を両方持つのか

> このドキュメントは「機能の使い方」ではなく「なぜこの形なのか」を残すためのもの。
> 実装の羅針盤。迷ったらここに立ち返る。消すのは簡単なので、熱いうちに書いておく。

## 一言で

**身体（実機）を動かす前に、頭の中（仮想デバイス）で動きを確かめる仕組み。**
それを組み込み H/W に対してやる。だから「物理機能の改善を、仮想で検証してから
実動作させる」ことができる。

## なぜ「新しい生物みたい」なのか

普通のソフトウェアは「本番で動かして、壊れたら直す」。Gapless Agent Runtime は、本番（実機）で
試す前に、自分の中（仮想デバイス）で予行演習する器官を持ち始めている。これは生物が
「運動を実行する前に頭の中で動作をシミュレートする」のと同じ構造。

```
        ┌─ 仮想デバイス（脳内シミュレーション）──┐
        │  「明るくしたらどうなる?」を先に試す    │
人間の   │              ↓ 安全に失敗できる         │
アイデア → 検証 ───────────────────────────────────→ 実機（身体）で実行
        │              ↑ --json で結果を観測       │
        └─ フィードバックで次のアイデアを補正 ────┘
```

| 普通のソフト | Gapless Agent Runtime |
|---|---|
| 本番で動かして、壊れたら直す | 仮想で予行 → 確信してから実機 |
| 状態は外から見るもの | 自分で状態を申告する（`gar sim env diag --json` / `ok`） |
| 人間が全部判断 | AI が試行 → 観測 → 補正のループを回す |

つまり「試す → 観る → 直す」という自己改善ループを、人間の外ではなくシステムの内側に
持つ。ただのツールは外部から操作されるだけだが、これは内部に予行演習と感覚を持つ。

## 構成要素を「器官」に対応させると

| 器官 | Gapless Agent Runtime の要素 |
|---|---|
| 想像力 / 内部世界モデル | 仮想デバイス（CUSE / gpio-sim / bridge） |
| 感覚・自己認識 | `gar sim env diag --json` の `ok` / exit code |
| 本能・行動規範 | `AGENT.md` の AI オペレーション契約 |
| 運動を実行する小脳 | AI（Codex / Copilot） |
| 意志（何をしたいか） | 人間 |

足りないピース: **仮想での予行結果を見て、実機に出す前に AI が自分で「やめておく / 直す」
判断をする閉ループ。** ここが繋がると、Gapless Agent Runtime は「指示されて動くツール」から
「試してから動く何か」へ変わる。

## この思想がもたらす次の方向（メモ、未実装）

「環境を整える / 状態を見る」の先にある「モノの動きをアイデアで直接いじる」レイヤー。

- 連続値・時間軸を持つ仮想 H/W モデル（今は ON/OFF 中心 → 明るさ・点滅速度などへ拡張）
- アイデアの言葉で叩く口（例: `gar hw set led18 --brightness 80` / `gar hw blink led24 --interval 200ms`）
- 自然言語のまま AI に投げ、AI が `gar hw ...` に翻訳 → `--json` で結果を確認して「明るくなりました」まで閉じる

これらは構想段階。YAGNI を守り、実際に欲しくなってから実装する。
ここに書いてあるのは「方向」であって「やること」ではない。

## なぜ妥協せずに作れるのか（前提）

このプロジェクトには納期も外部レビュアーもいない。唯一の不合格基準は「自分が気持ち悪いか」。
だから「`gar` に無い操作は生で叩かず `gar` に足す」のような、現場なら工数で却下される
正しさを最後まで通せる。後悔の溜まらないものづくりの場であることが、品質の源泉になっている。
