# 本質 — 3層をまたぐのではなく、あらゆる境界を1セッションに畳む

> このドキュメントは「機能の使い方」ではなく「このプロジェクトが本当は何の実証なのか」を残すためのもの。
> 視座の羅針盤。目の前の組み込み PoC に没入して、より大きな射程を見失いそうになったらここに立ち返る。

## 一言で

**Gapless Agent Runtime の最大価値は「開発・検証・実機の3層を同時に繋ぐこと」だと見えている。**
**だが本質はそこではない。「異種環境をまたぐたびに人間が払っていた"文脈の載せ替えコスト"を、**
**1つのエージェントセッションが連続性として吸収する」——その一般原理の実証である。**

3 という数にも、組み込みというドメインにも、本質的な必然性はない。
いま見えている「3層」は、その原理が**実際に動くことを示した最小事例（存在証明）**にすぎない。

## コアテクニックの正体（あえて身も蓋もなく言う）

この仕組みの土台は、特別な発明ではない。
**VS Code の Remote 開発（Remote-SSH / WSL / Dev Containers / Tunnels）＋ 統合ターミナル ＋ 単一の AI セッション**——
すべて公知・既存の機能の組み合わせである。

それでも価値がある。理由はこうだ。

> **「公知の機能であること」と「誰もその使い方をしていないこと」は両立する。**

実際、これらの機能を「異種環境を1セッションで横断作業しきる」ために束ねて使っている例は、
身近な開発現場にもほとんど存在しなかった。
つまり価値は **発明（invention）ではなく、適用の発見・転用（reframing）** の側にある。
枯れた土台の上で勝負することは、むしろ強い——安定し、無料で、学習コストが低く、サポートが続く。
**勝負は土台ではなく、土台の上に積む"環境を能力として束ね、縫い合わせる抽象"に宿る。**

## 「3層」を一段抽象化すると

```
  いま見えている形                       本質
  ┌─────────────────────┐        ┌──────────────────────────────┐
  │ 開発  ─ 検証 ─ 実機  │   →    │ N 個の異種環境を1セッションで  │
  │ (3層・組み込み限定)  │        │ 横断観測・操作・作業しきる     │
  └─────────────────────┘        └──────────────────────────────┘
        特定の一例                        一般原理（N も領域も自由）
```

同じ原理が刺さりうる先は、組み込みに限らない。

| ドメイン | "層" にあたるもの | 1セッションで繋ぐ価値 |
|---|---|---|
| 組み込み（現状） | 開発 / sim / 実機 | バイナリ透過の検証 |
| クラウド運用 | dev / staging / prod ＋ 監視 / DB / k8s | 障害調査〜修正〜デプロイを分断なく |
| データ / ML | ノートブック / 学習クラスタ / 推論本番 / データ基盤 | 実験から本番反映まで一気通貫 |
| 業務システム | 社内DB / SaaS API / オンプレ機器 / 帳票 | 部門をまたぐ作業を1人（1 AI）で |
| ロボティクス / IoT | sim / 実機フリート / クラウド管制 | 群を横断したオーケストレーション |

共通する真の価値は **「人間が環境の境界をまたぐたびに発生していた文脈の載せ替えを、エージェントが連続性として吸収する」** こと。
組み込み 3 層 PoC は、この原理が動くことを示した第 1 事例である。

## それでも冷静に置いておく2つの論点

熱に浮かされないために、弱点も同じ強さで書き残す。

1. **「VS Code の一機能」であることは、優位の持続性の弱さでもある。**
   誰でも真似できる。ゆえに防御可能な価値は機能そのものではなく、その上に積む
   **ドメイン固有の抽象とオーケストレーション**（各環境を能力として束ね、AI 運用契約・診断・シナリオで縫う部分）に宿る。

2. **「あらゆるところに接続できる」は諸刃。**
   汎用性の主張は魅力的だが、製品としては「どのドメインに最初に深く刺すか」を選ばないと、
   "何でもできるが何にも刺さらない" に陥る。
   汎用性を主張する権利は、N=3 組み込みの次に **別ドメインを 1 つ深くやり切る**ことで初めて獲得できる。

## だから、ここに立ち返る理由

目の前の作業はいつも「組み込みの 3 層」に見える。
だがこのプロジェクトが本当に証明しようとしているのは、もっと広い——
**「境界をまたぐ作業を、人間が往復するのではなく、エージェントが 1 セッションで畳みきれる」** という働き方そのものだ。

3 層は出発点であって、天井ではない。
