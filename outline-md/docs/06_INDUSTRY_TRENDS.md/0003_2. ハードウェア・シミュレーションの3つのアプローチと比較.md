## 2. ハードウェア・シミュレーションの3つのアプローチと比較

実機が存在しないクラウド環境で、いかにしてI2CやSPIといったハードウェア通信をシミュレートするか。業界では大きく3つの手法があります。

| アプローチ | 概要 | メリット | デメリット |
|---|---|---|---|
| **1. システムエミュレータ**<br>(Renode, QEMU等) | CPUからメモリ、周辺ペリフェラルまで全てを仮想マシンとして再現する。 | 非常に正確。OSのブートからテスト可能。 | 動作が重い。複雑なペリフェラルの設定・構築の難易度が極めて高い。 |
| **2. HAL層でのモック化** | C言語のコードレベルで `#ifdef SIMULATION` を入れ、ハードウェア操作関数をダミーに差し替える。（最も一般的） | 実装が簡単。ユニットテストに向いている。 | **「実機用バイナリ」と「テスト用バイナリ」が別物になる。** バイナリ互換性の担保ができない。 |
| **3. OS/device layer での互換 runtime**<br>(CUSE / gpio-sim / fake device) | **★本PoCのアプローチ★**<br>OSが提供する `/dev/*` インターフェースを実機互換に見せる。I2C/SPI は CUSE、GPIO は `gpio-sim` で提供する。 | **実機と同じ起動コマンド・同じ `/dev/*` 前提でテスト可能。** 高速。実機用起動定義を汚しにくい。 | Linuxカーネルの仕組み（ioctl等）の深い理解が必要で、実装難易度が高い。 |

**👉 本PoCの価値②：**  
本PoCが採用した CUSE / gpio-sim によるアプローチは、アプリケーション層（`sensor_demo.c`）の**実機互換性を保ちながら、システムエミュレータのような重さを排除した実用的な構成**です。
ただし、このプロジェクトの本当の焦点は「CUSE や gpio-sim を使うこと」自体ではありません。人手では採算が合いにくく、実機検証が始まると腐りやすい `/dev/*` 互換 runtime を、AI がプロジェクト専用に作り切り、保守し続けることです。

SPIやI2Cの複雑な `ioctl` リクエストを正確にパースして処理し、GPIO では `gpio-sim` の sysfs と Web Bridge を同期させる点は、AI エージェントが得意な「地味で構造化された互換実装」を活かす好例です。アプリ起動からシミュレーション固有の指定を外し、EC2 でも RasPi5 でも `~/sensor_demo` で起動できるところまで到達しました。
